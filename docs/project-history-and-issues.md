## 3. 캐싱/성능/트랜잭션/로깅 개선 단계

이 구간의 커밋들은 대부분 **아키텍처 품질과 운영 관점**에 초점을 맞추고 있습니다.

### 3.1 Redis 기반 캐싱 전략 고도화

- 관련 커밋
  - `CHORE : Redis 기반 캐싱 어노테이션 Aspect 로직 세부화 비즈니스 로직 개발`
  - `refactor: RedisConfig 개선 - 객체 직렬화 지원 추가`
  - `feat: CacheConfig 추가 - Redis 캐시 전략 구현`

**문제**
- Redis 캐싱 로직이 초기에는 단순/제한적이거나, 직렬화/역직렬화 전략이 명확히 정리되지 않았습니다.

**해결**
- `RedisConfig`를 개선하여 문자열용/객체용 템플릿을 분리
- `@CacheableResult` + `CachingAspect`로 비즈니스 로직 외부에서 캐싱을 제어
- `CacheConfig`에서 캐시 이름/TTL을 명시적으로 정의

**결과**
- 캐시 키/TTL 정책을 코드로 추적 가능하게 만들었고, 캐싱 실패 시에도 비즈니스 로직은 정상 동작하도록 안전하게 처리되었습니다.

### 3.2 Health 도메인 N+1 쿼리 및 쿼리 최적화

- 관련 커밋
  - `perf: HealthRecordRepository에 JOIN FETCH 쿼리 추가 - N+1 쿼리 문제 해결`
  - `refactor: HealthService 예외 처리 및 쿼리 최적화`

**문제**
- Health 도메인의 조회 로직이 다수의 연관 엔티티를 순차적으로 불러오면서 N+1 문제가 발생
  - 예: HealthRecord 목록을 가져온 뒤 Child/User를 lazy 로딩

**해결**
- Repository에 `JOIN FETCH` 쿼리를 추가하여 연관 엔티티를 한 번에 읽어오도록 수정
- Service 레이어에서 이러한 최적화된 메서드를 사용하도록 리팩토링

**결과**
- 데이터베이스 쿼리 수가 크게 감소하고, 특히 리스트 조회 API의 응답 시간이 개선되었습니다.

### 3.3 트랜잭션 경계 정리

- 관련 커밋
  - `refactor: HealthFacade에서 트랜잭션 어노테이션 제거`

**문제**
- Facade와 Service 양쪽에 `@Transactional`이 혼재되어, 실제 트랜잭션 경계가 불명확하고 불필요한 중첩이 발생할 수 있었습니다.

**해결**
- 트랜잭션은 Service 계층에만 적용, Facade는 호출/오케스트레이션 전담
- 읽기 전용은 `readOnly = true`, 쓰기 작업만 별도 트랜잭션으로 명시

**결과**
- 트랜잭션 경계가 단순해져 이해하기 쉬워졌고, 성능/락 경합 문제 가능성도 줄었습니다.

### 3.4 로깅 전략 및 MDC 기반 트레이싱

- 관련 커밋
  - `feat: logstash-logback-encoder 의존성 추가`
  - `feat: logback-spring.xml 추가 - 환경별 로깅 전략 구현`
  - `feat: LoggingUtil 유틸리티 추가 - MDC를 활용한 구조화된 로깅`
  - `refactor: LoggingAspect에 MDC 지원 추가`

**문제**
- 기존에는 텍스트 기반 로그에 request/사용자/메서드 컨텍스트가 일관되게 남지 않았고, 운영 환경에서 문제 추적이 어려웠습니다.

**해결**
- JSON 기반 구조화 로깅 도입 (`logstash-logback-encoder` + `logback-spring.xml`)
- `LoggingUtil`과 `LoggingAspect`에서 `MDC`를 사용해 `traceId`, `userId`, `childId`, `method`, `executionTime` 등 공통 필드 자동 주입

**결과**
- 로그만으로도 특정 요청/사용자/작업 흐름을 추적할 수 있게 되었고, 향후 ELK/모니터링 시스템과 연동하기 좋은 형태로 정리되었습니다.

## 4. 예외 처리/응답 표준화 및 테스트 도입

### 4.1 예외 처리 일관성 확보

- 관련 커밋
  - `feat: ErrorCode enum 추가 - 도메인별 에러 코드 체계화`
  - `feat: CareCodeException 기본 예외 클래스 추가`
  - `feat: 건강 관리 도메인 예외 클래스 추가`
  - `refactor: 기존 예외 클래스를 CareCodeException 상속으로 변경`
  - `feat: ErrorResponse 표준화된 에러 응답 DTO 추가`
  - `refactor: 전역 예외 핸들러 개선 및 표준화`

**문제**
- `IllegalArgumentException`, 커스텀 예외, 일반 RuntimeException 등이 섞여 사용되면서
  - HTTP Status/에러 메시지/로그 포맷이 일관되지 않음
  - 클라이언트에서 에러 타입을 구분하기 어려움

**해결**
- 모든 도메인 예외를 `CareCodeException` 기반으로 통일
- `ErrorCode` enum으로 에러 코드/기본 메시지 표준화
- `ErrorResponse`와 전역 예외 핸들러로 응답 포맷 단일화

**결과**
- API 에러 응답이 항상 같은 형태로 내려가고, 클라이언트/프런트엔드/모니터링에서 에러를 분석하기 쉬워졌습니다.

### 4.2 API 응답 래퍼 도입

- 관련 커밋
  - `feat: ApiResponse 표준화된 API 응답 래퍼 추가`

**문제**
- 각 컨트롤러가 서로 다른 응답 형태를 사용하면서, 클라이언트 입장에서 파싱/에러 처리 로직이 복잡해질 수 있음

**해결**
- `ApiResponse<T>` 클래스로 성공/실패 모두 동일한 스키마(`code`, `message`, `data`, `timestamp`) 사용

**결과**
- 도메인/엔드포인트에 관계없이 공통 응답 모델로 처리 가능해졌고, 문서화/테스트 코드에서도 이 패턴을 기준으로 삼을 수 있게 되었습니다.

### 4.3 테스트 코드 도입

- 관련 커밋
  - `test: HealthService 단위 테스트 추가`

**의미**

## 8. 정리: 이 프로젝트의 이슈/해결 흐름

1. **도메인 기능 우선 개발**
   - Health, User, CareFacility 등 핵심 도메인 기능을 먼저 구현
2. **DB/DTO/Mapper 정비**
   - ERD 문서화, DTO/Mapper 분리로 구조를 명확히 함
3. **성능/운영 이슈 대응**
   - Redis 캐싱, N+1 쿼리 해결, 트랜잭션 경계 정리
4. **예외/응답/로깅 표준화**
   - `CareCodeException` + `ErrorCode` + `ApiResponse` + JSON 로깅 + MDC
5. **테스트 및 문서화 시작**
   - HealthService 단위 테스트, 아키텍처 제안/완료 문서
6. **도메인 기능 고도화**
   - 챗봇 컨텍스트, 정책/알림/시설의 기능 확장

이 흐름을 통해, 프로젝트는 **“기능 위주의 초기 구현 → 운영/성능/품질 중심의 개선 → 안정화 후 기능 확장”**이라는 전형적인 성장 곡선을 밟고 있으며, 현재는 이미 상당히 정제된 아키텍처와 인프라를 갖춘 상태라고 볼 수 있습니다.
